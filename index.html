Argus — Architecture

A Solana smart-transaction stack. Core (Rust) = the eyes; Agent (TypeScript) = the judgment; the HTTP boundary is the architecture.

Live on mainnet Graded run: 15 sent / 6 failed / 9 landed 0.000135 SOL total Yellowstone gRPC · Jito bundles · OpenRouter Agent owns failure diagnosis

0Thesis

Argus — named for Argus Panoptes, the hundred-eyed watchman — observes the Solana network in real time, submits Jito bundles intelligently, tracks each submission across commitment levels, and delegates exactly one operational decision to an AI agent: failure diagnosis.

The one-sentence architecture Two services over HTTP/JSON. A Rust Core holds every deterministic, network-facing concern (streaming, leader window, bundles, tips, lifecycle, baseline classification, remedy execution). A TypeScript Agent holds a single Claude-powered judgment call. The process boundary is the literal embodiment of the challenge's required "clean separation between AI layer and core transaction stack."

The agent's mandate is deliberately narrow and deliberately un-scriptable: it reads the raw failure surface of a rejected transaction — the failing program, its structured instruction_error, and its program logs — and returns an open-ended Diagnosis, a Triage, and the executable Remedy. It is not a lookup over a fixed failure-class table; that table exists only as a baseline contrast the agent's reasoning is measured against (see §6 and ADR 0012).

1System Architecture

Argus is a monorepo of two runtimes deployed via docker-compose, talking only over HTTP/JSON. Rust earns its place on the Yellowstone gRPC firehose — tokio gives idiomatic bounded-channel backpressure and reconnection (the one genuinely hard infrastructure feature the challenge names). TypeScript hosts the agent for LLM-ecosystem ergonomics and fast prompt iteration. Neither leaks into the other: the agent holds zero transaction logic, the Core holds zero model logic. ADR 0001

Deployment view

  • Repo layout: core/ (Rust), agent/ (TypeScript), docs/ (ADRs + plan), logs/ (SQLite + JSONL + Markdown Lifecycle Log).
  • Real path runs on mainnet with SolInfra-sponsored credits and a dedicated low-balance keypair — Jito's Block Engine only lands bundles on mainnet, and judges verify slot numbers on explorers. Devnet is a destructive sandbox only. ADR 0002
  • One contract: Core → Agent is a single POST /decide; a scored run hard-gates on the agent's /health and refuses to start if it is down.

Agent (TypeScript) — the single judgment

  • index.ts — Express service: GET /health and POST /decide (zod-validated in, JSON out), default port 8787.

Storage — the Lifecycle Log is the deliverable

SQLite is the single source of truth; JSONL and Markdown are exports rendered from it. Two tables:

submissions

one row per attempt

run_id, attempt, nonce, bundle_id, signature, tip_lamports, submitted_at, landed_slot, processed_at, confirmed_at, finalized_at, failure_class · UNIQUE(run_id, attempt, nonce)

decisions

one row per agent decision

run_id, attempt, failure_class (baseline), remedy, baseline_remedy, diagnosis, triage, rationale, confidence, reasoning_trace, model, decided_at

A Run is identified by a prefix, not a column: the session is run-{ts} and payload k runs under child run_id = run-{ts}-p{k}, so (run_id, attempt) stays unique per payload with zero schema change. Run-scoped queries use WHERE run_id LIKE 'run-{ts}-%'. ADR 0011

3Data Flow Between Services

Happy path — submit → track → persist

The subscription opens before the bundle is submitted, so Inclusion is never missed. Tracking is bounded by a 60s timeout and reconciled afterward against getSignatureStatuses in case a Landed frame was dropped.

Lifecycle states

Failure path — diagnose → remedy → resubmit

A Jito bundle is all-or-nothing, so a faulted transaction never lands and leaves no on-chain meta.err. The one deterministic pre-submit source of {err, logs, unitsConsumed} is a preflight simulateTransaction — and that simulation output is the raw failure surface handed to the agent.

This keeps the network surface auditable and the build reproducible.

5Failure-Handling Strategy

Failure handling is the heart of the system — happy-path-only submissions are explicitly disqualified. Argus handles failure on two axes: a bounded four-class baseline (for remedy variation) and an unbounded program-error tail (for diagnosis variation).

The bounded baseline — four classes

A deterministic taxonomy, induced by fault injection and classified from the preflight simulation's structured instruction_error (not string-matching). Each maps to a default remedy that Core can execute on its own when the agent is unreachable.

Failure classInduced byDeterminismDefault remedy
Expired BlockhashSign against a real blockhash aged ~200 slots (past the ~150 window)Deterministicrefresh blockhash
Compute ExceededCU limit set to 1, below instruction needDeterministicraise CU limit (from re-simulated need)
Bundle FailureInclude a failing instructionDeterministicabort / rebuild
Fee Too LowTip below live floor under contentionProbabilisticbump tip

The unbounded tail — where a classifier goes blind

One identical malformed instruction ([0xff; 8], zero accounts) sent to three different real programs produces three distinct errors that the four-class baseline collapses to one blind bundle_failure → abort. This is the case that proves reasoning beats lookup.

Retry, recovery, and degradation

  • Remedy execution stays in Core. The agent names the remedy (refresh blockhash, bump tip, raise CU limit, hold-and-resubmit, abort); Core owns the magnitudes — e.g. the raised CU limit is derived from a max-CU re-simulation (×1.5, capped at 1.4M), not a tuned constant.
  • Attempt-2 is seeded clean. The injected attempt-1 cap applies only to attempt-1; the retry starts from a fresh state so a remedy is tested honestly.
  • Loud degradation. If the agent is unreachable within ARGUS_AGENT_TIMEOUT_SECS (~45s), Core falls back to the local baseline policy and records model="local-fallback" — the run never dies on an agent hiccup, and the fallback is visible in the log.
  • Stream resilience as failure handling. A dead endpoint is bounded by a reconnect ceiling that returns gave_up rather than hanging; a dropped Landed frame is recovered by post-stream RPC reconciliation.

6AI Agent Responsibilities

The agent owns exactly one operational decision — failure diagnosis (the challenge's "Failure Reasoning" option, with autonomous fault-injection recovery as a strict superset). It observes a failed transaction, reasons about why it failed, and decides what must change before retrying. Retry decisions come from the agent, not from hardcoded logic.

The contract

DirectionPayload
Core → Agent
POST /decide		attempt, error_text, instruction_error, failing_program_id, program_logs[], tip_lamports, tip_floor_p50/p75, blockhash_age_slots, cu_limit, cu_used, current_slot. No failure_class is sent.
Agent → Core
submit_decision tool		diagnosis (free text), triage (enum), remedy (enum), rationale, confidence — plus reasoning_trace (message.reasoning) and the serving model slug.

The agent reasons in three steps: diagnose (identify the failing program, decode its program-specific error from the logs), triage into one of four recovery buckets, then choose the executable remedy. The system prompt explicitly forbids a fixed error→remedy mapping.

Triage — the axis the agent reasons on

recoverable_by_refresh

e.g. expired blockhash → refresh & resubmit

recoverable_by_modification

e.g. compute exceeded → raise CU / bump tip

permanent

malformed call → abort (but with the program-specific reason)

funding

insufficient lamports → abort / top up
Why this isn't sequential automation Any decision specifiable cleanly enough to grade is encodable as a classifier — legible ⟹ enumerable ⟹ lookup-replicable. Handing the agent a four-class verdict and a five-element remedy set is a 4→5 mapping a match statement replicates: the "simple wrapper" the challenge disqualifies. The escape is not a cleverer decision — it's a different input: the unbounded, unstructured raw failure surface. An AMM alone defines its own custom-error enum (Custom(6022) differs per program and per version); a static classifier would need a combinatorial, perpetually-stale per-program table. Reasoning over the raw surface does not.

The honesty boundary: on a permanent failure the agent and the baseline both abort — the agent's added value there is the reason, not a different action. That candor is the point: the agent is graded on the diagnoses a lookup can't produce, not on theatrical disagreement. ADR 0012

The three required questions, answered from this run

Q1 — what does the processed→confirmed delta tell you? It is the cluster's vote-aggregation latency (≥⅔ of stake voting), i.e. consensus health — not inclusion speed. This run: 87–272 ms, median 123 ms across the 9 landed submissions — sub-quarter-second, i.e. healthy consensus. For contrast, the next hop confirmed→finalized took ~11.8–13.3 s (median ~12.2 s) because finalization waits for the slot to be rooted (~31 blocks).
Q2 — why never use a finalized blockhash for a time-sensitive tx? A blockhash is valid only ~150 slots (~60–90 s). A blockhash fetched at finalized commitment is already ~31 slots old the instant you receive it — ~20% of the window burned before you've built, signed, and submitted. Demonstrated directly: payload p0, injected with a blockhash aged 200 slots, was rejected at preflight with BlockhashNotFound; the agent diagnosed exactly that and recovery required a fresh blockhash.
Q3 — what if the Jito leader skips their slot? A Jito bundle is slot-specific and all-or-nothing. If that leader skips, the bundle is simply not included — it is not auto-forwarded to the next leader, and because tips pay only on inclusion, no tip is charged. The remedy is to resubmit to the next leader window with a fresh blockhash. This run is the receipt: all 6 faulted bundles were sent free (zero tip), and the two recoverable faults were resubmitted and landed.
